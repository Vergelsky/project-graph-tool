# ARCHITECTURE.md

# Project Execution Map

## Цель

Создать инструмент, автоматически строящий **карту выполнения проекта**, а не карту файлов.

Основная задача — помочь разработчику быстро понять архитектуру и бизнес-логику большого проекта без чтения тысяч строк кода.

---

# Основная идея

Большинство существующих инструментов показывают:

- структуру директорий;
- импорты;
- UML классов.

Для понимания проекта этого недостаточно.

Система должна строить граф **реального выполнения сценариев**.

Например:

```text
myapp.views.OrderViewSet.create  (trace root)
    ↓
OrderService.create()
    ↓
PaymentApi.charge()
    ↓
Order.save()
    ↓
PostgreSQL
```

Именно такие цепочки являются главным объектом визуализации.

---

# Основные принципы

## 1. Никакого ИИ при анализе

Построение структуры выполняется только детерминированными алгоритмами.

ИИ используется исключительно после построения графа для генерации текстовых описаний.

## 2. Главный объект — сценарий

Точки входа задаются **явно** — указатель на определение или вызов функции/метода:

- `trace --def path/to/file.py:42`
- `trace --def myapp.views.Handler.method`
- `trace --call path/to/file.py:88`

Список roots хранится в data-repo: `config/trace_queue.yaml`, обработанные — в `config/trace_done.yaml`.

От каждого root строится подграф вызовов; результаты объединяются в `output/execution_graph.json`.

## 3. Структура файлов вторична

Положение файла в проекте не является частью модели.

Модель строится только из логических зависимостей.

## 4. Единая графовая модель

Внутреннее представление системы — ориентированный граф.

Типы узлов:

- EntryPoint
- Controller
- Service
- Repository
- ORM
- Database
- External API
- Queue
- Cache

Типы связей:

- CALLS
- READS
- WRITES
- EMITS
- AWAITS
- RETURNS
- CREATES
- USES

# Используемые инструменты

## tree-sitter

Основной парсер исходного кода.

Используется для построения AST и извлечения структуры функций, классов, методов и вызовов.

---

## Jedi

Разрешение call sites через pipeline в `definition_resolver.py`:

1. tree-sitter извлекает вызовы (`CallInfo`, включая `callee_column`).
2. `goto(follow_imports=True)` и `infer()` на нескольких позициях call site.
3. Выбирается лучшее определение **внутри project_path** (не import line, не site-packages).
4. Для `Class.method` — fallback `resolve_class_method`.
5. BFS (`incremental_call_graph.py`) разворачивает только **expandable** callee (function/method).

Ограничения: динамические вызовы (`getattr`, signals, celery `.delay()`, часть Django Manager API) остаются `depends_on`.

---

## Pyright

Опциональное дополнение (заготовка в `pyright_bridge.py`). В hot path trace пока не используется.

---

## NetworkX

Основное внутреннее представление проекта.

Используется для хранения графа, поиска путей, анализа связности и подготовки данных к визуализации.

---

## Neo4j (опционально)

Используется только для очень больших проектов.

Позволяет выполнять сложные запросы к графу.

---

## Cytoscape.js

Интерактивная визуализация графа.

Поддерживает масштабирование, фильтрацию, поиск, группировку и различные раскладки.

---

## AI Agent

Получает готовый граф и код конкретного узла.

Формирует краткое описание назначения узла.

Не участвует в анализе архитектуры.

---

# Общий Pipeline

```
Python Project (project_path)
      ↓
Call graph: AST + Jedi (весь проект)
      ↓
Trace roots (явные указатели)
      ↓
BFS по call graph от каждого root
      ↓
Execution Graph merged (NetworkX)
      ↓
Cytoscape.js viewer
      ↓
AI Agent (описания узлов)
```

---

# Результат

Разработчик получает интерактивную карту проекта, позволяющую:

- увидеть полный путь выполнения сценария;
- быстро понять архитектуру;
- найти все пути до внешнего API;
- увидеть все записи в БД;
- изучать проект через граф, а не через файловую структуру.
