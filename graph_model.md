# GRAPH_MODEL.md

# Модель графа проекта

## Назначение

Данный документ определяет внутреннюю модель проекта.

Все остальные компоненты системы работают исключительно с этой моделью.

Парсеры наполняют её.

Визуализатор отображает её.

ИИ анализирует её.

Никакой компонент системы не должен зависеть от особенностей конкретного языка программирования.

---

# Основная идея

Любой программный проект представляется ориентированным графом.

Граф состоит из

- узлов (Nodes)
- связей (Edges)

Узел представляет некоторую логическую сущность.

Связь представляет отношение между двумя сущностями.

---

# Node

Каждый узел имеет следующую структуру

```json
{
    "id": "...",
    "type": "...",
    "name": "...",
    "qualified_name": "...",
    "description": null,
    "source_file": "...",
    "line_start": 0,
    "line_end": 0,
    "metadata": {}
}
```

---

## Поля

### id

Уникальный идентификатор.

Используется всеми компонентами системы.

---

### type

Тип узла.

Например

```
ENTRY_POINT
VIEW
SERVICE
REPOSITORY
MODEL
FUNCTION
METHOD
DATABASE
SQL
CACHE
QUEUE
EVENT
SIGNAL
EXTERNAL_API
FILE
MODULE
PACKAGE
```

---

### name

Короткое отображаемое имя.

Например

```
CampaignService
```

---

### qualified_name

Полное имя.

Например

```
apps.orders.services.OrderService.create
```

---

### description

Текстовое описание.

Заполняется ИИ.

На этапе построения графа всегда пустое.

---

### source_file

Файл определения.

Например

```
apps/orders/services/order.py
```

---

### metadata

Дополнительная информация.

Например

```
root_id

resolved_qualified_name

pointer_kind

call_graph_node_id

sql_table

celery_queue

external_resource

database

return_type

visibility

decorators
```

# Типы узлов

## ENTRY_POINT

Начало выполнения сценария — **явный trace root**.

Задаётся пользователем: определение или вызов функции/метода (`trace --def` / `trace --call`).

Примеры metadata

- `root_id`
- `resolved_qualified_name`
- `pointer_kind` (`def` | `call`)
- `call_graph_node_id`

# Граф сценариев

Основная единица анализа — сценарий выполнения.

Каждый сценарий начинается с ENTRY_POINT.

Например

```
root_id: create_order

↓

myapp.views.OrderViewSet.create

↓

OrderService

↓

OrderRepository

↓

Payment API

↓

Database
```

Все такие цепочки автоматически объединяются в общий граф проекта.

# Требования к модели

Модель должна быть полностью независимой от

- Python
- Django
- Cytoscape
- Neo4j
- tree-sitter
- Jedi
- ИИ

Любой компонент системы должен иметь возможность работать только через данный граф.

---

# Требования к расширяемости

Добавление нового языка программирования не должно требовать изменения модели.

Необходимо только реализовать новый парсер, который наполнит существующую модель графа.

---

# Главный принцип

**Граф является единственным источником истины.**

Исходный код нужен только для построения графа.

ИИ нужен только для описания графа.

Визуализация нужна только для отображения графа.

Все компоненты системы должны работать исключительно с этой моделью.
